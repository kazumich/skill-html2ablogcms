# html2ablogcms — 静的HTMLのお知らせを a-blog cms で動的化する

静的HTMLサイトの **「お知らせ」相当の繰り返しコンテンツ**だけを a-blog cms の動的コンテンツにするためのプレイブック。サイト全体のテーマ化ではなく「まず1本、更新情報をきちんと動的化する」ことに集中する。

**このスキルの肝は、特定のHTMLに対する正解コードを覚えることではない。** どんなマークアップでも通用する「①お知らせを見つける → ②データを a-blog cms 概念にモデリングする → ③機械的に変換する → ④人に管理画面作業を渡す」という**判断手順**を実行することにある。CSSフレームワーク（Tailwind/Bootstrap/独自）やマークアップの形（`ul/li`・`dl`・`table`・`article` 連打）が違っても同じ手順で適用する。

> 具体的な変換の通し例（Tailwind の飲食店サンプル）は [examples/worked-example.md](examples/worked-example.md) にある。**あれは一例**であって写経対象ではない。コード例のクラス名は元HTMLのものなので、実作業では**変換対象HTML自身のクラス・マークアップを保持**すること。
> 変数・ブロックの網羅的な一覧は、対象環境の `themes/system/acms-code/vars/` を読む（手順0）。公式ドキュメント: https://developer.a-blogcms.jp/document/ 。

## 考え方（メンタルモデル）

a-blog cms 側で「動かせない事実」はごく少ない。お知らせ動的化は要するに次の対応づけに尽きる：

- **起点は「格納カテゴリー」**。CMS化したいコンテンツ（テンプレ側で見つけた News / Service / 商品 …）は、必ず CMS 側の**1つのカテゴリー**に格納される。だから真っ先に決めるのは **「このコンテンツはどのカテゴリーに入るか（コード・名前）」**。
  - ⚠️ **テンプレ側にそのセクションがあること ≠ CMS(DB)側にカテゴリーがあること。** 対応カテゴリーが無ければ**作らないと**、モジュールIDの絞り込み先（cid）が空になり、何も出ない／全件出る。「Service を CMS 化して」と言われたら、まず**「その service を入れるカテゴリーはどれか／無ければ作る」**を確定する。
  - news は beginner にカテゴリーが既存だったので“不要”に見えただけ。**新種別（service 等）は格納カテゴリーが無いのが普通**なので、③モジュールIDより前に作成（人の作業）。
- **1つの更新情報 = 1エントリー**。日付・タイトル・カテゴリー・タグ・メインイメージ・本文（ユニット）を持つ。
- **表示は最大3つの「ビュー」**。それぞれ「カテゴリーで絞り込んだエントリー集合のクエリ」を 1モジュールIDが受け持つ：
  - トップの**抜粋**（最新数件）→ `Entry_Summary`
  - **一覧**（全件＋ページング）→ `Entry_Summary`
  - **詳細**（1件）→ `Entry_Body`
- 変換とは「**静的HTMLの繰り返しの literal 値を、上記モジュールの変数に差し替える**」作業。囲っているマークアップ（クラス・レイアウト）はそのまま残す。

CMS側の語彙（モジュール名・ブロック名・変数）は固定。あなたの仕事は、**目の前のHTMLがその語彙のどこに当たるかを判定**して機械的に置換すること。

> **モジュールID名は自由なラベル**。本スキルの例では `news_top` / `news_index` / `news_body` を使うが、これは「お知らせ＝news」に合わせて付けただけ。命名規約は `{種別}_top` / `{種別}_index` / `{種別}_body`。同じ手順を**商品なら `product_*`、サービスなら `service_*`、実績なら `works_*`** に置き換えてそのまま使える（中身は同じ `Entry_Summary` / `Entry_Body`、絞り込み先カテゴリーを変えるだけ）。**唯一の制約は、テンプレートの `id="…"` と管理画面で登録するモジュールIDの文字列を一致させること。** お知らせ以外のコンテンツに適用するときは、`SETUP.md`・テンプレ・カテゴリー名をその種別に読み替える。

## スコープ（やること / やらないこと）

- **やる**：お知らせ相当の更新情報の、存在するビューだけを動的化する（トップ抜粋／一覧／詳細のうち**あるものだけ**。3つ揃っているとは限らない）。
- **やらない**：お知らせ以外のページ（Home/About/Service/会社概要 等）は静的のまま据え置く。フォーム動的化（Formモジュール）やサイト全体のテーマ化はこのスキルの範囲外（別途対応する）。

## 最上位ルール — 人に依頼する範囲（DBは直接さわらない）

a-blog cms は**管理画面での設定を前提**に作られている。DB（`acms_*`）を SQL で直接書き換えたり、記事を SQL でシードしたりしない。整合性とキャッシュが壊れる。

| 作業 | 誰が |
|---|---|
| テンプレートファイルの生成・編集（HTML→従来タグ変換、`template.yaml`、head3行、管理BOX、include） | **Claude** |
| 管理画面作業の手順書 `SETUP.md` の生成（必要ならモジュールID用 YAML も。インポートは人） | **Claude** |
| ① **テーマの適用**（コンフィグ > テーマ設定） | **人** |
| ② **格納カテゴリーの作成**（CMS化する種別を入れる箱＝Entry_Summary の絞り込み先。例：`news` / `service` / `product`）。**既存と仮定しない**——テンプレにそのセクションがあっても、CMS に対応カテゴリーが無ければ作る | **人** |
| ③ **モジュールIDの設定**（条件・表示設定、または YAML インポート） | **人** |
| ④ **エントリー（記事）の投入**（静的HTMLの内容を実データ化） | **人** |

**順序依存**：①テーマ適用 → ②カテゴリー作成 → ③モジュールID → ④記事投入。**②が無いと③でカテゴリーを選べず cid 空＝絞り込み不能**（=「Service を動的化したのに何も／全件出る」の典型原因）。③が無いとテンプレに記事が出ない。
**「○○を CMS 化して」と頼まれたら、最初に「○○の格納カテゴリーはどれか／無ければ作る」を確定する。** news は beginner に既存だっただけで、新種別はまず②を用意する。
→ Claude の成果物は「テンプレート一式」＋「①〜④を人が進める `SETUP.md`」。**人の作業を勝手に実行しない**。

## 前提環境

- a-blog cms が **beginner テーマでインストール済み**の状態から始める。
- **管理用パーツの多くは `system` テーマに fallback がある**。a-blog cms は「アクティブテーマ → system テーマ」の順でテンプレートを解決するので、新テーマに無くても `@include`/`@extends` は system 側で解決される。**まず存在を確認し、fallback があるものはコピーしない**（無駄な複製を避ける）：
  - `admin/action.html`（管理BOX）/ `admin/entry/action.html`（詳細用）/ `_layouts/unit.html` … 通常 **system にあるのでコピー不要**。そのまま `@include("/admin/action.html")` 等で使える。
  - `admin/entry/field.html`（カスタムフィールド入力欄）/ `admin/entry/field-seo.html`（SEO欄）… system に**無い**ことが多い。**カスタムフィールドや SEO 欄を使うときだけ** beginner からコピーする。
  - 本文ユニット表示は、新テーマに **`include/unit.html` を1行 `@extends("/_layouts/unit.html")` だけで作れば足りる**（個別カスタムが要るときだけ `@section` で上書き）。
  - 判断手順：使いたい include について `<docroot>/themes/system/...` に同名ファイルがあるか確認 → あれば `@include` で使う／無ければ beginner からコピー。
- テーマは `<docroot>/themes/<name>/` に作る。docroot は環境により異なる（`htdocs/` / `web/` / サイトルート直下 など）。まず対象環境の docroot を確認してからパスを組み立てる。**稼働中の既存テーマを改造せず、静的HTMLから新テーマを作る。**

## 着手前の前提確認（不明点は実装前にまとめて聞く）

テンプレを書き始める前に、**次の確認事項を点検し、ユーザー指定で埋まっていない／環境から判断できないものは、決め打ちせずまとめて質問する**。「○○を CMS 化して」だけでは足りない前提が必ずある。ここを飛ばすと「カテゴリーが無くて cid 空」のような手戻りになる。**未確定のまま実装に入らない**（特に #2）。

| # | 確認すること | 既定/調べ方 | 不明なら聞く |
|---|---|---|---|
| 1 | **動的化する対象**（どのセクション/種別か） | HTMLから推定（手順1） | 候補が複数／曖昧なら確認 |
| 2 | **格納カテゴリー**（コード・名前。既存か新規か） | CMSに該当カテゴリーがあるか確認 | **無い／未指定なら着手をいったん止めて確認**：どのカテゴリーを使うか、無ければ「`code=…` で作成してよいか」を提案して聞き、**作成を依頼→確定してから実装に入る**（手順1.5。＝最重要ゲート） |
| 3 | **作るビュー**（トップ抜粋／一覧／詳細のどれ） | HTMLに存在するもの | 一覧や詳細が無いサイトもある→確認 |
| 4 | **デザイン分岐**（他カテゴリーと一覧/詳細の見た目が違うか／今後別種別もCMS化するか） | — | 置き場所（直下 or `<コード>/`）が変わる→確認 |
| 5 | **適用テーマ名 / docroot** | `ddev describe` 等 | 不明なら確認 |
| 6 | **区分ラベルの正規化**（カテゴリー／サブカテゴリー／タグ／カスタムフィールドのどれにするか） | 手順2の振り分けルール | 判断が割れるなら確認 |
| 7 | **管理作業は人が行う前提**（②③④） | 本スキルの大原則 | 基本は確認不要（SETUPで渡す） |

→ **埋まっていない項目を1回でまとめて質問**してから着手する（小出しにしない）。特に **#2 は最頻出の落とし穴**。

## 手順

### 0. リファレンスを読む（推測でタグを書かない）

使うモジュールの変数表を必ず先に読む。a-blog cms は `themes/system/` に全モジュールのリファレンスを同梱しているので、対象環境の docroot を頭に付けて読む：

```
Read: <docroot>/themes/system/acms-code/vars/Entry_Summary.html      # 一覧で使える変数・ブロック
Read: <docroot>/themes/system/acms-code/vars/Entry_Body.html         # 詳細で使える変数・ブロック
Read: <docroot>/themes/system/acms-code/snippets/entry_summary.html  # 完成形スニペット
Read: <docroot>/themes/system/admin/config/entry/summary_body.html   # config_key と選択肢
Read: <docroot>/themes/system/admin/config/entry/body_body.html
```

`themes/system/admin/config/{ns}/{sub}_body.html` には、そのモジュールの設定項目（HTMLの `name="…"` が `config_key`、`<option value>` が取りうる値）が全部載っている。SETUP.md に書く設定値はここで裏取りする。

### 1. お知らせを「構造」から特定する（マークアップ非依存）

決め打ちせず、次のシグナルで探す。**CSSフレームワークやタグ種別に依存しない**ことが重要：

- **見出し/区切り**：`News` / `お知らせ` / `新着情報` / `トピックス` / `topics` / `info` / `更新情報` / `whats new`。`id`/`class`/見出しテキストのいずれでも。
- **繰り返し構造**：「日付 ＋ タイトル（＋カテゴリ/タグ ＋リンク）」の塊が**複数並ぶ**箇所。`<li>`連打でも、`<dl><dt><dd>`でも、`<table>`の行でも、`<article>`連打でも同じ。
- **一覧への導線**：「一覧へ」「もっと見る」「View all」リンク。
- **詳細ページ群**：日付や連番のファイル（`/news/20250807.html`、`/info/123.html` 等）が集まったディレクトリ。

特定したら、**存在するビューだけ**を確定する（3つ揃っている必要はない）：

| ビュー | 典型 | 変換後（テーマ直下の規約名） | モジュール | 無い場合 |
|---|---|---|---|---|
| トップ抜粋 | トップ内の更新情報セクション | `_top.html` 内（サイトトップ） | `Entry_Summary` | トップに無ければ作らない |
| 一覧 | `/news/` 等の全件ページ | `index.html`（汎用一覧テンプレ） | `Entry_Summary` | 一覧ページが無いサイトもある |
| 詳細 | 1記事1ページ | `_entry.html`（汎用詳細テンプレ） | `Entry_Body` | 詳細が同一ページ内アンカー/モーダルなら、CMS化では独立した詳細ページになる旨を人に伝える |

> **テンプレートの置き場所＝2択（`template.yaml` は常に直下の規約名のまま。ディレクトリは絶対に書かない）。** a-blog cms は `/news/` `/service/` を**カテゴリーコードから自動で**テンプレに解決するので、置き場所だけで出し分ける：
>
> | 状況 | 一覧/詳細の置き場所 |
> |---|---|
> | 動的カテゴリーが1つ、または**全カテゴリーで一覧/詳細の見た目が同じ** | テーマ直下 `index.html` / `_entry.html`（汎用） |
> | **カテゴリーごとに見た目が違う**（例：news と service でレイアウト別） | **コード名ディレクトリ**：`news/index.html`・`news/_entry.html`、`service/index.html`・`service/_entry.html`（**自動で優先**）。直下の `index.html`/`_entry.html` は汎用フォールバックとして残す |
>
> ⚠️ **先読みが肝**：いま動的化するのが1カテゴリーでも、**後で別カテゴリーを“別デザイン”で CMS 化する予定があるなら、最初から `<コード>/index.html`・`<コード>/_entry.html` に置く**。直下に置いてしまうと、後から `news/` へ移す手戻りが出る。`_top.html`（サイトトップ）だけは常に直下。<br>（本スキルの sample は news 単独だったので直下 `index.html`/`_entry.html` で足りたが、service も別デザインで足すなら本来は `news/` 配下が正。）

**判断に迷う / お知らせが見つからない / 複数候補があるときは決め打ちせずユーザーに確認する**（どのセクションを動的化するか）。

#### 1.5 格納カテゴリーを確定する（ビューより先・必須）

ビューを実装する前に、**「特定したコンテンツを CMS のどのカテゴリーに格納するか」を必ず確定**する（メンタルモデルの“起点”）。判定フロー：

1. **ユーザーが格納カテゴリーを指定しているか？** 指定があり、それが CMS に存在する → それを使う。
2. **指定が無い／CMS にも該当カテゴリーが無い** → **ここで実装をいったん止めてユーザーに確認する（ブロッキング・ゲート）**。
   - 例：「『Service』を動的化するには格納先のカテゴリーが要ります。`code=service` / 名前=Service で**作成をお願いできますか**？（既存を使うならどれか教えてください）」のように、**提案コード・名前を添えて聞く／作成を依頼する**。
   - **作成を依頼し、使うカテゴリー（コード）が確定してから実装に入る**。決め打ち・後回しで進めない。黙って進めると、③モジュールIDで絞り込み先を選べず **cid 空＝何も出ない／全件出る**。**テンプレ側に Service セクションがあること ≠ CMS にカテゴリーがあること。**
3. 確定したカテゴリーは**人の②作業**。SETUP にも明記（コードはURL／テンプレ配置と揃える＝`service` など）。

> **「カテゴリーが無い／未確定のままテンプレ実装を始めない。」** 無ければ「どうするか聞く・作成を依頼する」→確定、を**着手の前提**にする。news は beginner に `news` カテゴリーが既存だったため確認を飛ばしても動いてしまっただけ。新種別は必ずこのゲートを通す。

補足：一覧/詳細モジュールは `cid_scope`/`eid_scope` を `global` にすれば**モジュール設定に cid を書かずに**URLコンテキストで絞れる（手順5・references参照）。ただし**カテゴリー自体は必要**（記事の格納先＆ `/service/` のURL解決のため）。**トップ抜粋だけは cid 固定が必須**で、その格納カテゴリーは存在必須。

### 2. データをモデリングする（← 汎用化の本体）

特定した繰り返し1件から「1記事が持つ情報」を洗い出し、**各情報を a-blog cms のどの概念に載せるか**を判定する。マッピングの“正解表”を覚えるのではなく、次の**振り分けルール**で決める：

| 静的HTMLに現れる情報 | a-blog cms 概念 | 判定ルール |
|---|---|---|
| 投稿日（`<time>`・テキストの日付・ファイル名の日付） | **投稿日** `{date#…}` | エントリー標準。日付がタイトル内テキストにしか無くても、投稿日として入力してもらう |
| 記事タイトル | **タイトル** `{title}` | エントリー標準 |
| 記事へのリンク先 | **URL** `{url}` | CMSが自動採番。静的の `20250807.html` 等は捨てて `{url}` に置換 |
| 概要・リード文 | **概要** `{summary}` | 一覧/抜粋用。Body側 `fulltext` を on |
| サムネ/アイキャッチ画像 | **メインイメージ** `{path}`（`BEGIN_IF [{path}/nem]`） | 各記事に1枚の代表画像なら標準のメインイメージ |
| 区分ラベル（少数の固定集合：新商品/お知らせ/メディア 等） | **カテゴリー** `categoryField`>`{fieldCategoryName}` | 「種類分け・絞り込み・URLになる」＝カテゴリー。数が限られ排他的ならこれ |
| フリーな付箋的キーワード（#夏メニュー 等、多数・自由） | **タグ** `tag:loop`>`{name}` | 自由語・複数付与・横断検索したいならタグ |
| 上記で表せない属性（バッジ色名・外部リンク・価格 等） | **カスタムフィールド** | 標準で表せないものだけ。`admin/entry/field.html` に追記、設定は SETUP.md |
| 詳細ページの本文（見出し・段落・画像・引用…） | **ユニット**（`unit:veil`>`@include unit.html`） | 本文は人がユニットで入力する。**静的HTMLの本文マークアップは「見本」で、テンプレには残さない**（テンプレはユニットを描画するだけ） |

**重要な汎用ポイント**：
- 区分が「色違いのバッジ」でも、見るべきは**意味**（種類分けか？自由語か？）。クラス名や色は CSS 側に残し、CMS では category/tag に正規化する。
- 詳細本文は literal を変数化するのではなく、**ユニット表示に置き換える**。静的の `<p>…</p>` 群はサンプル記事の中身であって、テンプレートには持ち込まない。

### 3. 変換アルゴリズム（マークアップの形に依らない）

> **変換前に退避（バックアップ）**：上書き・リネームするファイル（front page `index.html`→`_top.html`、一覧の `index.html` や `<コード>/index.html`）は、着手前に**同じ場所へ `*_backup.html` を複製**する（例 `_top_backup.html` / `index_backup.html` / `service/index_backup.html`）。変換後に**元デザインと見比べて取りこぼし（交互レイアウト等）を検出**でき、失敗時に戻せる。**ディレクトリは作らない／最後に掃除**。新規作成する `_entry.html` は元の静的詳細（`catering.html` 等）が残るのでバックアップ不要。

一覧系（トップ抜粋・一覧）はどんなマークアップでも次の手順で機械的に変換する：

1. **繰り返し各件を見比べ、「変数に置き換える値」以外が全件で同一かを確認する**（← 今回のミス防止の要）。
   - **値の差**（日付・タイトル・URL・画像・概要 …）＝変数化する対象。
   - **値ではない差**（例：偶数番目だけ `md:flex-row-reverse`／先頭だけ大きい／行ごとの背景色）＝**位置依存のバリエーション**。1件に畳むと消えるので、畳まず **`{oddOrEven}`（odd/even）・連番 `{i}`・出力が兄弟要素になるのを利用した CSS `:nth-child()`** で再現する。**向きは必ず元と照合**（`oddOrEven` は1件目=odd）。
     例：`<article class="md:flex-row<!-- BEGIN_IF [{oddOrEven}/eq/even] --> md:flex-row-reverse<!-- END_IF -->">`
2. 繰り返しの**最小単位**を1つだけ残し、残りの兄弟は削除する。
3. その単位を `<!-- BEGIN entry:loop -->` … `<!-- END entry:loop -->` で囲む。
4. 単位の中の **literal 値だけ**を変数に置換する（日付→`{date#…}`、見出し→`{title}`、`href`→`{url}` …）。**囲っているタグ・クラス・属性はそのまま残す**。位置依存のクラスは手順1で決めた `{oddOrEven}` 等の出し分けにする。
5. ループの外側（同じモジュール内）に、空表示 `<!-- BEGIN notFound -->…<!-- END notFound -->` と、必要ならページャー `<!-- BEGIN pager:veil -->…<!-- END pager:veil -->` を足す。
6. 全体を `<!-- BEGIN_MODULE Entry_Summary id="…" -->` … `<!-- END_MODULE Entry_Summary -->` で囲む。

最小の中立例（フレームワーク非依存。`dl` でも `table` でも同じ考え方）：

```html
<!-- 変換前（静的・3件並んでいる） -->
<ul class="news">
  <li><a href="/news/20250807.html"><time>2025.08.07</time><span>新メニュー</span></a></li>
  <li><a href="/news/20250801.html"><time>2025.08.01</time><span>リニューアル</span></a></li>
  <li>…</li>
</ul>

<!-- 変換後（1件をloop化。class="news" 等の元マークアップは保持） -->
<!-- BEGIN_MODULE Entry_Summary id="news_top" -->
<ul class="news">
  <!-- BEGIN notFound --><li>お知らせはまだありません。</li><!-- END notFound -->
  <!-- BEGIN entry:loop -->
  <li><a href="{url}"><time datetime="{date#Y}-{date#m}-{date#d}">{date#Y}.{date#m}.{date#d}</time><span>{title}</span></a></li>
  <!-- END entry:loop -->
</ul>
<!-- END_MODULE Entry_Summary -->
```

詳細ページ（`Entry_Body`）は、記事1件分の `<article>` を `entry:loop` で包み、**本文部分だけはユニット表示に差し替える**（手順2参照）。

### 4. a-blog cms 固有の“動かない作法”（ここは仕様。守る）

マークアップが何であれ、次は固定ルール：

- **`<head>` に3行**（無いと管理ボタン等が崩れる）：
  ```html
  <link rel="stylesheet" href="/css/acms-admin.min.css">
  <script src="%{JS_LIB_JQUERY_DIR}jquery-%{JS_LIB_JQUERY_DIR_VERSION}.min.js" charset="UTF-8"></script>
  <!-- BEGIN_MODULE Js --><script src="%{ROOT_DIR}acms.js{arguments}" charset="UTF-8" id="acms-js"></script><!-- END_MODULE Js -->
  ```
- **管理BOX**を全ページ統一位置に：`<!-- BEGIN_MODULE Touch_Login --><div>@include("/admin/action.html")</div><!-- END_MODULE Touch_Login -->`（詳細ページは `admin/entry/action.html`）。
- **`template.yaml`**（テーマ直下）。**6種すべて書き、ディレクトリは書かない**（beginner と同形）。トップは `_top.html` にリネームする（`tpl_top` と `tpl_index` が同じ `index.html` で衝突しないように）：
  ```yaml
  tpl_top    : _top.html
  tpl_index  : index.html
  tpl_detail : _entry.html
  tpl_404    : 404.html
  tpl_admin  : admin.html
  tpl_login  : _member-admin/login.html
  ```
  - `404.html` / `admin.html` / `_member-admin/login.html` の実体は通常 **system テーマに fallback がある**ので、テーマに置かなくても 6 行書いてよい（無い環境のときだけ beginner からコピー）。
  - **カテゴリーのディレクトリ（`news/` 等）は template.yaml に書かない。** `/news/` も `/service/` も、カテゴリーコードから上記の汎用テンプレへ自動解決される。特定カテゴリーだけ専用テンプレにしたいときに `<コード>/index.html` や `<コード>/_entry.html` を置けば自動優先される（これも yaml には書かない）。
- **カテゴリー名は入れ子で出す**：`<!-- BEGIN categoryField:veil --><!-- BEGIN categoryField -->{fieldCategoryName}<!-- END categoryField --><!-- END categoryField:veil -->`（`{categoryName}` 直書きはしない）。表示制御は **`category_on`／`category_info_on`（既定ON）**。`category_field_on` ではない。
- **画像の存在判定**：`<!-- BEGIN_IF [{path}/nem] -->…<img src="%{ROOT_DIR}{path}[resizeImg(600)]">…<!-- END_IF -->`。
- **概要の文字数調整**：`{summary}[trim(120,'…')]`。
- **ページャーは `pager:veil` で囲む**（最終ページで「もっと見る」が自動的に消える）。
- **本文ユニット**：`<!-- BEGIN unit:veil -->@include("/include/unit.html")<!-- END unit:veil -->`。
- **`<title>` 動的化**：`<title>%{ENTRY_TITLE} | サイト名</title>`。

> これらブロックの全変数・全 config_key は手順0の `vars/` と `admin/config/.../*_body.html` で確認する。Entry_Summary / Entry_Body の完成形は同梱の `snippets/` と [examples/worked-example.md](examples/worked-example.md) にある。

**番号ページャー**（「もっと見る」ではなく 1・2・3… を出したい場合）は `pager:veil` の中をこの形にする：

```html
<!-- BEGIN pager:veil -->
<!-- BEGIN backLink --><a href="{url}">前へ</a><!-- END backLink -->
<!-- BEGIN page:loop -->
<span {pageCurAttr}[raw]><!-- BEGIN link#front --><a href="{url}"><!-- END link#front -->{page}<!-- BEGIN link#rear --></a><!-- END link#rear --></span>
<!-- END page:loop -->
<!-- BEGIN forwardLink --><a href="{url}">次へ</a><!-- END forwardLink -->
<!-- END pager:veil -->
```

ページ番号は `link#front` / `link#rear` で囲む方式（`link` / `span` ブロックは無い）。`{pageCurAttr}[raw]` で現在ページに属性（class等）が付く。

### 5. 人への手順書 `SETUP.md` を生成する

テーマと一緒に渡す管理画面作業の手順書を、テーマ外（プロジェクト側）に1ファイル作る。**最上位ルールの①〜④を人が順にこなせる**ように、実際に作ったビューの分だけ書く。骨子：

```
## 0. テーマの適用 … コンフィグ > テーマ設定で「<name>」に切替
## 1. 格納カテゴリーを作成（最重要・③より前）… 対象種別のカテゴリーが無ければ必ず作る。コードはURL/テンプレ配置と揃える（news / service など）。バッジ等を区分するならサブカテゴリーも。※既存があるかは事前確認済みのはず
## 2. モジュールIDを作る（③として下記フォーマットで詳述）
## 3. 記事を投入 … 静的HTMLの各記事を、フィールド対応表どおりにカテゴリーへ登録
## 動作確認 … / にトップ抜粋, /news/ に一覧＋もっと見る, 記事クリックで詳細
```

**モジュールIDの章は手作業ミスが最も多い。次の“事故らせない”書式で必ず書く：**

1. **最初に「識別子＝テンプレの `id` と完全一致」を太字で警告**し、`識別子 / 使うモジュール(サマリーor本文) / テンプレの記述箇所` の対応表を置く（コピペできるように識別子をそのまま記載）。1文字違うと「他カテゴリ混在／空」になる、と明記。
2. **モジュールIDごとに見出しを立てて縦に積む**（`### <識別子> — 説明` → `#### 条件設定`（表）→ `#### 表示設定`（表））。表は `項目 / 設定値`、表示設定には **`効くテンプレ変数`列**を足して「なぜONにするか」を結びつける。`—`=設定不要・`OFF`=明示オフ。**モジュールIDが増えても下に積むだけ**で済む（横並びマトリクスにしない＝列が増えて破綻するため）。
3. 各モジュールの表の下に **補足をリストで**書く（注意点と、「**確認：どのURLで何が出れば成功か**」を1項目）。
4. 章末に **完了チェックリスト**（識別子スペル一致／サマリーと本文の取り違え／`datetime-desc`／一覧は「ページのURLコンテキストON」＋「ページャーON」両方／混在時は識別子不一致を疑う）。

設定値の要点（裏取りは手順0の `*_body.html`）：
- **cid は「固定」か「URLコンテキスト任せ」か**を、ビューの置き場所で決める（重要）：
  - **一覧（カテゴリーURL `/service/` に出る）** → **カテゴリーの `cid_scope=global`（cid 明示しない）** が原則。URLのカテゴリー文脈で自動絞り込み。**カテゴリー未確定でも設定でき、複数カテゴリーで使い回せる**。表示順=**`datetime-desc`**、ページングは「引数ページのURLコンテキストON」＋「ページャーON」。
  - **詳細（エントリーURLに出る）** → カテゴリー空＋**エントリー `eid_scope=global`**。URLの eid で記事決定。
  - **トップ抜粋（サイトトップ `/`＝カテゴリー文脈なし）** → **cid を明示（`cid_scope=local`）必須**。URLから決められないため。→ その**格納カテゴリーは存在必須**（着手前の前提確認 #2）。
- 一覧で特定カテゴリーに固定したいとき（global を使わない場合）だけ、カテゴリー絞り込み＋必要なら「下層を含むON」を指定。
- 表示ON項目とテンプレ変数の対応：`{summary}`→概要(fulltext)、`{path}`→画像(image_on)、`tag:loop`→タグ(tag/tag_on)、ページャー→pager_on。
  - ⚠️ **`categoryField`／`{fieldCategoryName}`（カテゴリー名バッジ）は `category_on`（本文は `category_info_on`）＝どちらも既定ON が駆動する。`category_field_on` では駆動しない**（あれは「カテゴリーに付けたカスタムフィールド」用の別物）。実機エクスポートで `category_field_on: off` でもバッジが出ることを確認済み。
- **基本は人が管理画面で設定**する（②③④）。手順書には管理画面の項目名で書けば足りる。
- ただし **モジュールID は YAML 生成→人がインポート、も実用的に可能**（環境非依存）。**生成するときは関連モジュール（top/index/body 等）を必ず1ファイルにまとめる**（モジュールごとに分割しない＝人のインポートが1回で済む）。構造・インポート時の自動再マップ・**変数駆動の ON/OFF（性能最適）**・cidスコープ設計・落とし穴は **[references/module-id-yaml.md](references/module-id-yaml.md)** に集約した。**インポート自体は必ず人が行う**。

### 6. 検証チェックリスト ＆ よくある崩れ

- [ ] 動的化したのは**お知らせ相当のビューだけ**（他ページは静的のまま）
- [ ] 使うモジュールの `vars/` を読んでから変数を書いた（推測で書いていない）
- [ ] `entry:loop` は繰り返し最小単位を1件だけ残して loop 化（兄弟は削除）／**元のクラス・マークアップを保持**した
- [ ] **畳む前に各件を見比べ、変数化する値“以外”が同一か確認**した／位置依存の差（交互レイアウト等）は `{oddOrEven}`/`:nth-child` で再現した
- [ ] 変換前に `*_backup.html` を退避し、**変換後の表示を元（backup）と見比べた**（取りこぼし無し）。確認後はバックアップを掃除
- [ ] 区分は意味で判定し category/tag/カスタムフィールドに正規化した（色やクラスはCSS側に残す）
- [ ] 詳細本文はユニット表示に置換した（静的の本文マークアップをテンプレに残していない）
- [ ] head 3行 ／ 管理BOX ／ `template.yaml` がある
- [ ] `categoryField` の入れ子、ページャーの `pager:veil`、`datetime-desc` を守った
- [ ] **格納カテゴリーを確定した**（無ければ②で作成を促した。テンプレにあるからCMSにもある、と仮定しない）
- [ ] `SETUP.md` に ①テーマ ②カテゴリー ③モジュールID ④記事投入 を順番つきで書いた
- [ ] DB を SQL で直接さわっていない／記事を SQL でシードしていない

| 症状 | 原因 | 対処 |
|---|---|---|
| 管理ボタン/フォームが崩れる | `acms-admin.min.css` 未読込 | head 3行 |
| **モジュールIDでカテゴリーを選べない／cid が空** | 絞り込み先カテゴリーが**未作成**（新種別は格納カテゴリーが無いのが普通。テンプレ側の有無とは別） | **一覧/詳細なら `cid_scope=global`（＋詳細は `eid_scope=global`）にしてURLコンテキスト任せ**にすれば cid 不要で回避。**トップ抜粋は cid 固定が必須**なので②でカテゴリーを先に作る（③より前） |
| **他カテゴリの記事まで出る／対象以外が混ざる** | モジュールID未登録で「既定設定＝ブログ全体の最新」が効いている（テンプレ自体は正常） | SETUP.md ③でモジュールIDを作り、カテゴリー＝対象＋必要なら下層含むON |
| お知らせが1件も出ない | カテゴリーに記事が無い／絞り込み先カテゴリー違い | SETUP.md ②④を人に依頼／カテゴリー紐付け確認 |
| 並びが日付降順にならない | order値ミス | `datetime-desc` |
| 最終ページでも「もっと見る」が残る | `pager:veil` で囲っていない | veilで囲む |
| 概要/画像が空 | Body側configがoff | `fulltext`/`image_on` を on |
| 記事URLが静的のまま | リンクを`{url}`にしていない | `href="{url}"` |
